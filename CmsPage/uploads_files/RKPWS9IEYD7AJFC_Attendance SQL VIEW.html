SQL VIEW – Help Document.

How to Use the SQL Views

New SQL

  1. New SQL –    Create a New SQL View,
    • UID is the unique Id of SQL View UID is the Enterable Text,
    • API name should be name of the SQL view. API name should not be allowed in the special character
    • API and API method always standard format of REST And POST.
  2. Db Connection:   Db connections name should be selected which we have mentioned in DB Connections
  3. Schema - Based on the Db connection permitted schema name should be showed.
  4. URL – Content Management System (CMS) URL should be mentioned
  5. SQL – SQL Queries should be entered, SQL can get much more complex depending on the requirements of your database Tables.
  6. Save – Save the entries
  7. SQL Views – Saved SQL Views are showed in the format of API Name, API Method, UID, SQL Method is having two methods like SQL or Group, Action
  8. Action
    • 8.1 Edit – Edit Should be modify the SQL only
    • 8.2 API End Point
      • 8.2.1 Body Params – SQL is having any parameters it should be display, mention the data type in Body params.
      • 8.2.2 API Docs – This is the combination of API URL and Body params you can utilize the API for Front end developer
    • 8.3 RUN SQL – Run SQL is the result set of the SQL in Column and Row format.
  9. Group SQL- Group SQL is the combination of SQL, Each SQL is having own parameters which that parameters are associate the Group body params. Each SQL result set one below one will display in Rest method
  10. New Group SQL - Similar as the SQL it has unique Id; Group API name should be enterable and not allowed the special character. Db Connection and schema name will be displayed in the API Studio connections, Schema option, URL is the CMS page URL will be entered here
  11. Select SQL – Multiple SQL you can Select depending on the Schema.

 

 

 

 

 

 

 

 

Attendance - API

Scenario:  Biometric attendance systems utilize unique biological characteristics such as fingerprints, Biometric data is unique to each individual data punch captured in Local Server which this data manually restores in the Server (API Studio Attendance Database) in Public Schema. The data collection should be converted into readable format using API Studio SQL View.

Why Attendance API: Biometric system not able to give the Breaking Time report, So We tried in the SQL View Attendance API. SQL View is having a Two Tables are Employee master and Iclock Transaction Table. Iclock transaction is every day biometric punching data, employee master is the details of the employee.

SQL Columns - Employee Code, First Name (Employee Name), Punch Date, Punch Time, Sl. No, Attendance Status.

Sl. No & Attendance Status

0 is the Absent how to identify the absent an employee does not punch in via a biometric attendance system

1 is the Check in of Attendance Status how to identify this status an employee punch in between 06:00 am to 10:30 am in case of multiple punch is available Minimum time will be taken.

1.1 is Half an Hour Late employee punching time in between 10:31 am to 12:59, In case of multiple punch is available maximum time will be taken

2 is the Lunch Breaking Log Out employee punching time in between 01:00 to 01:30 In case of multiple punch is available maximum time will be taken

3 is the Lunch Breaking Log In employee punching time in between 01:31 to 02:00 In case of multiple punch is available maximum time will be taken

3.1 is the Lunch Breaking Log in Late employee punching time in between 02:01 to 03:00 In case of multiple punch is available maximum time will be taken

3.2 is Half Day Leave last punch is the in between 02:01 to 02:30 and not a checkout in between 06:00 to 10:00pm, it should not be lunch breaking log in late

4 is the Checkout employee punching time in between 06:00 to 10:00 pm In case of multiple punch is available maximum time will be taken

4.1 is the Early Checkout   employee punching time in between 03:01 to 05:59 pm In case of multiple punch is available maximum time will be taken
